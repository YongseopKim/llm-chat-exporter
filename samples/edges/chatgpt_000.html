타당한 부분

유행을 전부 따라잡는 전략은 실패하기 쉽다

AI 도구는 교체 비용이 낮고 변화가 빨라서, 새 도구를 많이 아는 것이 곧 생산성을 의미하지 않습니다. 특히 프레임워크를 학습하는 시간과 실제 문제를 푸는 시간은 경쟁 관계입니다.

다음 질문은 유용합니다.

  • 이 기술이 어떤 실제 실패를 해결하는가?

  • 현재 방식보다 성공률·비용·지연시간을 얼마나 개선하는가?

  • 제거하거나 교체하기 쉬운가?

  • 3개월 뒤 없어져도 시스템이 유지되는가?

다만 글에 제시된 “2년 후에도 중요한가?”는 지나치게 먼 기준일 수 있습니다. 스타트업이나 실험에서는 앞으로 3~12개월간 유효하고 교체 가능하다면 충분히 좋은 선택일 수 있습니다.

평가와 관측성을 먼저 구축하라는 조언

글에서 가장 중요한 대목입니다.

에이전트 시스템의 문제는 데모가 작동하는지가 아니라 다음을 아는 데 있습니다.

  • 어떤 입력에서 실패하는가

  • 같은 입력에 다시 성공하는가

  • 모델을 바꿨을 때 개선됐는가

  • 비용 증가가 성공률 증가를 정당화하는가

  • 잘못된 행동을 실행 전에 막을 수 있는가

따라서 tracing, 실제 실패 사례, 골든 데이터셋, 회귀 테스트를 일찍 갖추라는 조언은 합리적입니다. 단, 첫날부터 거대한 평가 플랫폼을 만들 필요는 없습니다. 초기에는 수십 개의 대표 사례와 구조화된 로그만 있어도 충분합니다.

단일 에이전트에서 시작하라는 조언

이것도 대체로 옳습니다. 멀티에이전트는 다음 비용을 추가합니다.

  • 상태 동기화

  • 역할 중복

  • 오류 전파

  • 비용과 지연시간 증가

  • 재현성과 디버깅 난이도 증가

실제 병목이 하나의 컨텍스트로 처리할 수 없는 업무 분리, 병렬 탐색, 권한 격리 등에 있을 때만 여러 에이전트를 도입하는 편이 낫습니다. 멀티에이전트는 기본 구조가 아니라 관찰된 실패에 대한 처방이어야 합니다.

모델을 교체 가능하게 만들라는 조언

타당하지만 “모델은 완전히 상품화되므로 아무 모델이나 바꿔 끼울 수 있다”는 뜻으로 받아들이면 안 됩니다.

모델마다 다음이 다릅니다.

  • 도구 호출 형식과 안정성

  • 지시 준수 방식

  • 장문 컨텍스트 성능

  • 병렬 호출 행동

  • 안전 정책

  • 지연시간과 가격

  • 실패했을 때의 양상

따라서 올바른 목표는 모델을 완전히 추상화하는 것이 아니라, 모델 변경을 평가셋으로 검증할 수 있게 만드는 것입니다.

과장되거나 잘못 일반화된 부분

“7가지 복리 개념” 중 일부는 영구적인 원리가 아닙니다

컨텍스트 관리, 도구 인터페이스, 평가, 상태 관리, 격리 실행은 지속성이 높은 개념입니다.

하지만 다음은 구분해야 합니다.

  • MCP는 구체적인 프로토콜입니다.

  • Orchestrator–Subagent는 여러 아키텍처 중 하나입니다.

  • 생각–행동–관찰 루프는 특정 에이전트 실행 패턴입니다.

  • 파일 시스템 상태는 상태 표현 방식 중 하나입니다.

MCP는 현재 AI 애플리케이션과 외부 데이터·도구 연결을 표준화하려는 개방형 프로토콜이며 상당한 채택 가능성이 있습니다. 그러나 프로토콜을 사용한다고 인증, 권한 관리, 멱등성, 트랜잭션, 감사 로그가 자동 해결되는 것은 아닙니다. Model Context Protocol+2Model Context Protocol+2

즉, 장기적으로 남는 것은 “MCP 자체”라기보다 다음 원리일 가능성이 더 큽니다.

모델과 도구 사이의 계약을 명시적으로 정의하고, 권한과 입출력을 검증하며, 구현을 교체 가능하게 만드는 것.

특정 프레임워크를 전면 추천·배제하는 목록은 신뢰하기 어렵다

“LangGraph는 프로덕션 기본”, “AutoGen·CrewAI·Semantic Kernel·DSPy는 절대 건너뛰어라” 같은 문장은 기술 평가라기보다 취향에 가깝습니다.

특히 DSPy는 멀티에이전트 프레임워크라기보다 LLM 프로그램을 선언하고 최적화하는 접근에 가깝기 때문에, 다른 도구와 한 묶음으로 배제하는 것부터 분류가 부정확합니다.

프레임워크는 이름으로 결정할 것이 아니라 다음 조건으로 평가해야 합니다.

  1. 실패 시 실행을 재개할 수 있는가

  2. 상태와 도구 호출을 재현할 수 있는가

  3. 관측성을 외부 시스템으로 내보낼 수 있는가

  4. 내부 추상화를 우회할 수 있는가

  5. 종속성을 제거할 때 핵심 비즈니스 로직이 남는가

조건을 충족한다면 직접 작성한 단순 루프도 괜찮고, 프레임워크도 괜찮습니다.

“샌드박스는 필수”도 범위가 너무 넓습니다

코드 실행, 브라우저 조작, 파일 수정, 외부 명령 실행이 있다면 샌드박스나 이에 준하는 격리는 사실상 필수에 가깝습니다.

하지만 읽기 전용 검색, 문서 분류, 제한된 내부 API 호출만 수행한다면 별도의 VM 샌드박스보다 다음이 더 중요할 수 있습니다.

  • 최소 권한

  • 도구별 허용 목록

  • 스키마 검증

  • 쓰기 작업 승인

  • 요청별 예산 제한

  • 감사 로그

“샌드박스”라는 제품 범주보다 권한 경계와 피해 반경 제한이 본질입니다.

커리어에 관한 부분은 가장 경계해야 합니다

“22살 신입과 35살 선임이 같은 출발선”은 사실이 아닙니다

AI가 코드 작성, 검색, 초안 생성 속도의 격차를 줄이는 것은 맞습니다. 하지만 숙련자의 핵심 가치는 코드 타이핑 속도가 아닙니다.

숙련자는 보통 다음에서 차이를 만듭니다.

  • 문제가 잘못 정의됐음을 감지

  • 요구사항 간 충돌 발견

  • 데이터와 평가 기준 설계

  • 보안·법률·운영 리스크 예측

  • 실패 시 복구 전략 수립

  • 조직적 제약하에서 의사결정

  • 유지보수 비용 판단

  • AI가 낸 그럴듯한 오류 식별

AI는 구현 능력의 일부를 평준화하지만, 판단력·책임·도메인 지식까지 평준화하지 않습니다. 오히려 코드 생산 비용이 낮아질수록 잘못 만든 시스템의 양도 늘어나므로, 무엇을 만들지와 무엇을 배포하지 않을지 판단하는 능력은 더 중요해질 수 있습니다.

전통적 커리어 패스가 “무너졌다”기보다 변형되고 있습니다

학위나 직급만으로 능력을 증명하기 어려워지고, 공개된 결과물과 실제 문제 해결 기록이 중요해지는 흐름은 타당합니다.

그러나 포트폴리오가 다음을 대체하지는 못합니다.

  • 대규모 시스템 운영 경험

  • 장기간의 팀 협업

  • 규제 산업 지식

  • 사고 대응 경험

  • 고객·이해관계자와의 조정 능력

  • 타인의 작업을 검토하고 책임지는 경험

앞으로 유리한 사람은 “신기술을 가장 많이 아는 사람”도, “무조건 빨리 출시하는 사람”도 아닙니다.

문제를 좁게 정의하고, 결과를 측정하고, 빠르게 출시하되, 실패 비용을 제한하며 반복할 수 있는 사람에 가깝습니다.

이 글에서 실제로 가져갈 실행 원칙

제가 다시 쓴다면 다음 여섯 가지로 정리하겠습니다.

  1. 기술이 아니라 업무 지표를 먼저 정한다.
    예: 상담 처리시간, 계약 검토 누락률, 주문 입력 오류율.

  2. 가장 단순한 워크플로부터 시작한다.
    에이전트 없이 한 번의 모델 호출과 구조화된 출력으로 해결되면 그것이 더 낫습니다.

  3. 모든 실패를 저장한다.
    실패 입력, 도구 호출, 결과, 인간 수정 내용을 회귀 평가셋으로 전환합니다.

  4. 관찰된 실패만큼만 복잡도를 추가한다.
    계획기, 메모리, 하위 에이전트, 검색, 샌드박스를 미리 모두 넣지 않습니다.

  5. 프레임워크보다 경계를 소유한다.
    프롬프트, 도구 스키마, 상태, 평가 데이터와 비즈니스 로직을 특정 프레임워크에 묻지 않습니다.

  6. 학습 시간을 고정하고 제작 시간을 보호한다.
    예를 들어 정보 탐색 10%, 실제 구축·사용자 관찰·평가 90%처럼 제한합니다.

가장 중요한 수정은 이것입니다.

“학습하지 않는 능력”이 필요한 것이 아니라, 무엇을 지금 배우지 않을지 결정하는 능력이 필요합니다.

학습과 실행은 반대말이 아닙니다. 좋은 학습은 현재 제품의 구체적인 실패에서 시작하고, 바로 실험과 평가로 돌아갑니다. 나쁜 학습은 실패나 고객과 무관하게 새 도구의 문법만 수집하는 것입니다.

확신도: 높음. 핵심 실무 원칙에 대한 판단은 비교적 확실합니다. 다만 원문의 작성자와 전체 출처가 제시되지 않아, 특정 도구를 배제한 근거가 원문에는 별도로 있었는지는 확인하지 못했습니다.
